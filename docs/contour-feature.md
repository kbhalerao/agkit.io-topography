# Contour lines ŌĆö feature spec

Generate 2 ft contour lines (with 10 ft index contours) for the field area,
extending ~50 ft past the field boundary so technicians can see how contours
cross the fence. Primary consumer: the erosion app's slope-shooting UI,
where the technician draws a representative slope profile over the contour
basemap.

Not yet implemented. Tracked separately from the SQS-trigger migration
(which is complete).

## Visual contract

- **Minor contours** every 2 ft, thin lines on the map.
- **Index contours** every 10 ft, thicker lines.
- Both come from a single contour pass ŌĆö index = `level_ft % 10 == 0`.
- 2 ft from a 10 m DEM is over-fit (DEM vertical accuracy is ~1ŌĆō2 m); the
  *relative* spacing is honest, the absolute precision is not. Same pipeline
  becomes meaningful when fed a LiDAR DEM (future premium offering).

## Buffer & clip

- Lambda reuses `clipped_raster` from `build_common_cache()` ŌĆö 220 m
  bbox-buffered DEM, gives r.contour edge stability for free.
- Output contour lines are clipped to `field_shp.buffer(50 ft)`. The buffer
  must be applied in a planar CRS (3857 or local UTM), not in degrees ŌĆö
  50 ft is not a degree distance and varies with latitude.
- No HUC-12 clip. (HUC-12 was a deferred idea for auto length-slope; that
  workflow is being replaced by a technician-drawn profile, so HUC-12 is no
  longer needed.)

## Lambda side ŌĆö `agkit-topography`

New method `LambdaGISProcessor.contour_lines(self, payload)` in
`app/geoworker.py`:

1. `self.build_common_cache(payload)` ŌĆö reuses 220 m-buffered DEM.
2. `r.mapcalc dem_ft = clipped_raster * 3.28084` ŌĆö USGS DEM is in meters;
   contour level values stored in **feet** to match RUSLE2 consumers.
3. Single `r.contour input=dem_ft step=2 levels=auto` pass ŌĆö generates all
   2 ft contours in one shot.
4. `v.out.ogr format=GeoJSON` to dump the contour vector.
5. Post-process the GeoJSON:
   - Tag `is_index = (level_ft % 10 == 0)` on every feature.
   - Rename the level attribute to `level_ft`.
6. Reproject `field_shp` ŌåÆ planar CRS, `buffer(50 ft)`, reproject back to
   4326, use as a cutline to clip the GeoJSON features.
7. Post back via `Poster.post_raster(...)` (piggyback the existing raster
   endpoint ŌĆö the magic-link auth machinery is what we want to reuse). File
   extension `.geojson` is the routing signal on the receiving side.

New color ramp: none ŌĆö vector output, no colorize step.

Add to `AWS_LAMBDA_EVENT_CONFIGURATION["public_elev_10m"]["jobs"]`:

```python
{"lambda_function": "contour_lines",
 "post_back": {"file_extensions": ("geojson",),
               "parameter": "contours_public_10m",
               "layer": "USGS 10m Contours"}},
```

### `models.py`

New proxy following the `SoilProperties` / `TopographyLayer` pattern from
`tier1apps/gislayers/models/proxy_models.py`:

```python
from tier1apps.gislayers.models.base_models import GISVectorLayer
from tier1apps.gislayers.exceptions import ContentTypeInvalidError

class TopographyVectorLayer(GISVectorLayer):
    """
    Proxy of GISVectorLayer for DEM-derived vector products (contour lines).

    One row per (Field, parameter). Generated by the agkit-topography Lambda
    and POSTed back to the topography postback endpoint.

    Convention:
      * `name` formatted as "USGS <layer>" (e.g., "USGS 10m Contours").
      * `geometry` holds a GeometryCollection of LineStrings (one per
        contour line, 4326).
      * `document` carries `{features: [{level_ft, is_index}, ...]}` ŌĆö
        parallel array of per-feature properties matching `geometry`.
    """
    def clean(self):
        super().clean()
        if not isinstance(self.content_object, Field):
            raise ContentTypeInvalidError(
                "TopographyVectorLayer must be attached to a Field"
            )

    class Meta:
        proxy = True
```

### `services.py` ŌĆö postback upsert helper

New `upsert_topography_vector(field, parameter, layer_name, filename, file_obj)`
parallel to the existing `upsert_topography_raster`. Reads the uploaded
GeoJSON, splits into:

- `geometry`: `GeometryCollection([LineString, ...])` from `feature.geometry`
- `document`: `{"features": [{"level_ft": ..., "is_index": ...}, ...]}` from
  `feature.properties`, **same order as the geometry collection**

Upserts the `TopographyVectorLayer` row keyed on `(field, name, metadata__parameter)`.

### `api_views.py` ŌĆö dispatch

Rather than building a separate endpoint, extend `RasterPostbackView` to
inspect the uploaded file's extension. `.geojson` ŌåÆ `upsert_topography_vector`,
else ŌåÆ `upsert_topography_raster` (current behavior). One endpoint, two
destinations. The magic-link auth is unchanged.

(Alternative considered: separate `vector-postback` endpoint. Rejected because
the auth machinery and URL minting are identical and would just be duplicated.
Single-endpoint dispatch keeps `LambdaEventBuilder` simpler too ŌĆö same
`_raster_post_url` works.)

## Frontend side ŌĆö `agkit.io-ui` (note for future work)

- New MapLibre vector source pulled from the `TopographyVectorLayer` ORM row.
- Two paint layers off the same source: `filter: ["==", ["get", "is_index"], false]`
  for thin minor contours, `filter: ["==", ["get", "is_index"], true]` for
  thick index contours.
- Slope-profile UI: technician draws a polyline over the contours; frontend
  samples the elevation raster (or the contours themselves) to compute slope
  along the line. Pure client-side, no Lambda call.

## What we deliberately didn't do

- **No adaptive contour count** ("at least N lines"). Tried it before; not
  stable across terrain types.
- **No 1 ft contours.** Honest floor on 10 m DEM is 2 ft; finer than that
  is noise.
- **No smoothing.** Stay honest to the source. Smooth client-side at render
  time if a viewer prefers it.
- **No HUC-12 clip.** Superseded by the technician-drawn profile workflow.
- **No deprecation of the L/LS scalar reducers.** They stay as the sensible
  starting default; the technician-drawn profile is the authoritative
  override, not the only path.
